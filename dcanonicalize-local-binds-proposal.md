# Core golden-file tests are fragile due to unstable local binder names

## Summary

`-ddump-simpl`-based golden tests often fail for reasons unrelated to semantics: GHC changes the names of local binders, and the dumped Core changes even when the optimization result is effectively the same.

Add an opt-in flag, `-dcanonicalize-local-binds`, that canonicalizes local term binders in pretty-printed Core dumps. This should make dump-based tests and manual inspection more robust without changing the optimization pipeline, generated code, or interface files.

## The problem

Tests that compare `-ddump-simpl` output against golden files break easily. Local binder names such as `go1`, `go2`, `x1`, `ds`, `eta`, `ipv` are generated by internal naming heuristics (desugaring, tidying, SpecConstr, etc.) and can change after seemingly unrelated compiler modifications.

`-dsuppress-uniques` does not fully solve this:

- System names can still expose uniques
- Even when uniques are hidden, the base `OccName` itself can change

Example of the fragility:

```core
fusionElemMap
  = \ x eta ->
      joinrec {
        go1 ds
          = case ds of { ... }
```

can become:

```core
fusionElemMap
  = \ x eta ->
      joinrec {
        go2 ds1
          = case ds1 of { ... }
```

The semantics are the same, but the golden test fails.

This is not hypothetical. In the discussion on `!15812`, a concern was raised that full-Core golden files are useful but fragile exactly because names like `go1` can change spuriously. And `#27101` asks for checking the desired compiler output more directly than today's coarse grep-based test.

## Why existing workarounds are unsatisfying

Today, some tests avoid full golden files and instead use `grep_errmsg` to look for a few patterns in Core output. That helps with binder-name churn, but it loses most of the benefits of golden tests.

`#18909` documents the ergonomics problem:

- failures do not show useful diffs
- `--test-accept` does not reliably refresh these tests
- `.stderr` files can drift away from the compiler's real output
- the test only checks a pattern, not the surrounding Core that explains the optimization

In the recent comments on `#18909`, it was noted that there are roughly 100 such tests with stale `.stderr` files, and that converting them to real golden files would be preferable if the dumps were stable enough.

## Proposal

Add a new debug flag:

```text
-dcanonicalize-local-binds
```

When enabled, GHC canonicalizes local term binders in pretty-printed Core dumps.

The flag is:

- opt-in
- output-only
- backwards compatible when disabled

## Precise scope

The proposal should be deliberately narrow.

### Goals

- Make Core dumps more stable for tests
- Make Core dumps easier for humans to compare across compiler changes
- Preserve the parts of the dump that are semantically interesting: top-level names, imported names, constructors, primops, structure, and control flow

### Non-goals

- Do not make all dump output byte-for-byte stable
- Do not change generated code, interface files, unfoldings, or optimization behavior
- Do not rename top-level binders or imported/external names
- Do not solve every source of churn in dumped Core

## Canonicalization rules

The intended behavior is:

- Canonicalize only local term binders with internal/system names
- Leave external names unchanged
- Leave top-level binders unchanged
- Start fresh for each top-level binding
- Assign canonical names in left-to-right binding-site order
- Keep dead binders as `_`

The traversal order should follow binding sites, because that matches how humans read the dump:

- lambda binders: left to right
- `let`/`join`/`letrec`/`joinrec` binders: source order in the binding group
- case binders and pattern binders: alternative order, then binder order within the alternative

One simple naming scheme is:

```text
clbA, clbB, ..., clbZ, clbA1, clbB1, ...
```

The exact prefix is not especially important. The important part is that it is deterministic, obviously synthetic, and visually distinct from real user-written names.

## Example

With:

```text
-ddump-simpl -dsuppress-all -dsuppress-uniques -dcanonicalize-local-binds
```

we would prefer:

```core
fusionElemMap
  = \ clbA clbB ->
      joinrec {
        clbC clbD
          = case clbD of {
              [] -> False;
              : clbE clbF ->
                case clbA of { I# clbG ->
                case clbE of { I# clbH ->
                  case ==# clbG (+# clbH 1#) of {
                    __DEFAULT -> jump clbC clbF;
                    1# -> True
                  } } }
            }; } in
      jump clbC clbB
```

Each top-level binding is canonicalized independently.

## Why this should happen at dump time

A testsuite-only normalizer is possible, but it is weaker than a compiler-supported dump flag.

Doing this in GHC has several advantages:

- it helps humans reading dumps, not only tests
- one mechanism works for all Core-dump consumers
- tests can opt in without maintaining ad hoc text-processing pipelines
- it avoids implementing a fragile parser for Core syntax in the testsuite driver

In other words, this is better viewed as a dump-formatting feature than as a testsuite hack.

## Implementation direction

The proposal should avoid committing too early to an unnecessarily invasive implementation.

The key requirement is:

- canonicalization must happen only on the dump-rendering path

That means this should be an output normalization step applied immediately before pretty-printing Core, not a transformation that re-enters the optimization pipeline.

There are at least two plausible implementation strategies:

1. A printer-oriented approach that threads a renaming environment while rendering Core.
2. A lightweight pre-print alpha-renaming step applied only to the bindings being dumped.

Either can work. The proposal should specify the observable behavior, not overspecify the internal mechanism.

## Expected impact

This would make it much easier to write and maintain tests that check actual Core rather than grep fragments.

Concrete benefits:

- the `elem`/fusion tests discussed in `!15812` and `#27101` can use full-Core goldens with less churn
- some `grep_errmsg`-based Core tests could be converted to proper golden tests, addressing the workflow problems in `#18909`
- more dump-based tests become practical, which supports the broader goal in `#21983` of improving `-ddump-*` flag coverage

This does not eliminate all instability in Core dumps, but it removes one of the most common and least informative sources of noise.

## Open questions

I think the proposal should answer these explicitly:

1. Should the first version canonicalize only term binders?

Recommendation: yes. Type/coercion binders are less often the source of test fragility, and many dumps already suppress most of that information.

2. Should canonicalization reset for each top-level binding?

Recommendation: yes. It keeps dumps readable and localizes diffs.

3. Should the flag apply only to Core dumps?

Recommendation: yes initially. The motivating examples are all about Core, and other IRs can be considered separately later.

## Related GitLab issues and discussions

- `!15812`: discussion around improving `elem` tests; concern raised that full Core goldens are valuable but currently fragile because of unstable local names such as `go1`
- `#27101`: asks for checking the desired compiler output more directly in `T17752`
- `#18909`: `grep_errmsg` has poor diff ergonomics; later comments note stale `.stderr` files and suggest moving toward real golden files
- `#21983`: `-ddump-*` flags lack test coverage; more stable dumps reduce the cost of adding such tests

## Bottom line

The proposal is worthwhile, but it should be framed as a narrowly scoped dump-output normalization feature:

- useful for tests
- useful for humans
- opt-in
- Core-only to start with
- semantics-preserving because it only changes rendering

That framing makes the motivation clearer and avoids the impression that we are proposing a new optimization pass just to make golden files happier.
