# Core golden-file tests are fragile due to unstable local binder names

## Summary

`-ddump-simpl`-based golden tests often fail spuriously: GHC changes local binder names (`go1`, `go2`, `x1`, `ds`, `eta`, `ipv`) even when the optimization result is the same. Add `-dcanonicalize-local-binds` to rename these to deterministic names (`clbA`, `clbB`, ...) at dump time.

## The problem

Local binder names are generated by internal heuristics (desugaring, tidying, SpecConstr) and can change from unrelated compiler modifications. `-dsuppress-uniques` doesn't fully help — System names still expose uniques, and the base `OccName` itself can change.

Example: this Core

```core
fusionElemMap
  = \ x eta ->
      joinrec {
        go1 ds
          = case ds of { ... }
```

could become:

```core
fusionElemMap
  = \ x eta ->
      joinrec {
        go2 ds1
          = case ds1 of { ... }
```

Semantics identical, golden test fails.

This came up in `!15812` where full-Core golden files were proposed but flagged as fragile. `#27101` asks for checking Core output more directly than today's grep-based tests.

## Why grep_errmsg isn't a good workaround

About 100 tests use `grep_errmsg` instead of full golden files — a workaround for binder-name churn. But as `#18909` notes:

- failures don't show useful diffs
- `--test-accept` doesn't reliably refresh these tests
- `.stderr` files can drift from real compiler output
- you only see matched patterns, not the surrounding Core

With stable dumps, these tests could convert to proper golden files.

## Proposal

Add `-dcanonicalize-local-binds` to canonicalize local term binders in Core dumps:

- Rename local binders with internal/system names to `clbA`, `clbB`, ..., `clbZ`, `clbA1`, `clbB1`, ...
- External names, top-level binders, constructors, primops stay unchanged
- Each top-level binding starts fresh (`clbA`, `clbB`, ...)
- Dead binders stay as `_`
- Opt-in, output-only, backwards compatible

The canonicalization happens at dump-rendering time — no change to optimization, generated code, or interface files.

## Example

With `-ddump-simpl -dsuppress-all -dsuppress-uniques -dcanonicalize-local-binds`:

```core
fusionElemMap
  = \ clbA clbB ->
      joinrec {
        clbC clbD
          = case clbD of {
              [] -> False;
              : clbE clbF ->
                case clbA of { I# clbG ->
                case clbE of { I# clbH ->
                  case ==# clbG (+# clbH 1#) of {
                    __DEFAULT -> jump clbC clbF;
                    1# -> True
                  } } }
            }; } in
      jump clbC clbB
```

## Why in GHC, not just the testsuite

A testsuite-only normalizer would need to parse Core syntax, which is fragile. Doing this in GHC:

- helps humans reading dumps, not just tests
- works for all Core-dump consumers
- tests opt in with a single flag, no ad hoc text-processing
